9 * WHAT IS EBUCKETS?
10 * -----------------
11 * ebuckets is being used to store items that are set with expiration-time. It
12 * supports the basic API of add, remove and active expiration. The implementation
13 * of it is based on rax-tree, or plain linked-list when small. The expiration time
14 * of the items are used as the key to traverse rax-tree.
15 *
16 * Instead of holding a distinct item in each leaf of the rax-tree we can aggregate
17 * items into small segments and hold it in each leaf. This way we can avoid
18 * frequent modification of the rax-tree, since many of the modifications
19 * will be done only at the segment level. It will also save memory because
20 * rax-tree can be costly, around 40 bytes per leaf (with rax-key limited to 6
21 * bytes). Whereas each additional item in the segment will cost the size of the
22 * 'next' pointer in a list (8 bytes) and few more bytes for maintenance of the
23 * segment.
24 *
25 * EBUCKETS STRUCTURE
26 * ------------------
27 * The ebuckets data structure is organized in a hierarchical manner as follows:
28 *
29 * 1. ebuckets: This is the top-level data structure. It can be either a rax tree
30 * or a plain linked list. It contains one or more buckets, each representing
31 * an interval in time.
32 *
33 * 2. bucket: Each bucket represents an interval in time and contains one or more
34 * segments. The key in the rax-tree for each bucket represents low
35 * bound expiration-time for the items within this bucket. The key of the
36 * following bucket represents the upper bound expiration-time.
37 *
38 * 3. segment: Each segment within a bucket can hold up to `EB_SEG_MAX_ITEMS`
39 * items as a linked list. If there are more, the segment will try to
40 * split the bucket. To avoid wasting memory, it is a singly linked list (only
41 * next-item pointer). It is a cyclic linked-list to allow efficient removal of
42 * items from the middle of the segment without traversing the rax tree.
43 *
44 * 4. item: Each item that is stored in ebuckets should embed the ExpireMeta
45 * struct and supply getter function (see EbucketsType.getExpireMeta). This
46 * struct holds the expire-time of the item and few more fields that are used
47 * to maintain the segments data-structure.
48 *
49 * SPLITTING BUCKET
50 * ----------------
51 * Each segment can hold up-to `EB_SEG_MAX_ITEMS` items. On insertion of new
52 * item, it will try to split the segment. Here is an example For adding item
53 * with expiration of 42 to a segment that already reached its maximum capacity
54 * which will cause to split of the segment and in turn split of the bucket as
55 * well to a finer grained ranges:
56 *
57 * BUCKETS BUCKETS
58 * [ 00-10 ] -> size(Seg0) = 11 ==> [ 00-10 ] -> size(Seg0) = 11
59 * [ 11-76 ] -> size(Seg1) = 16 [ 11-36 ] -> size(Seg1) = 9
60 * [ 37-76 ] -> size(Seg2) = 7
61 *
62 * EXTENDING BUCKET
63 * ----------------
64 * In the example above, the reason it wasn't split evenly is that Seg1 must have
65 * been holding items with same TTL and they must reside together in the same
66 * bucket after the split. Which brings us to another important point. If there
67 * is a segment that reached its maximum capacity and all the items have same
68 * expiration-time key, then we cannot split the bucket but aggregate all the
69 * items, with same expiration time key, by allocating an extended-segment and
70 * chain it to the first segment in visited bucket. In that sense, extended
71 * segments will only hold items with same expiration-time key.
72 *
73 * BUCKETS BUCKETS
74 * [ 00-10 ] -> size(Seg0)=11 ==> [ 00-10 ] -> size(Seg0)=11
75 * [ 11-12 ] -> size(Seg1)=16 [ 11-12 ] -> size(Seg1)=1 -> size(Seg2)=16
76 *
77 * LIMITING RAX TREE DEPTH
78 * -----------------------
79 * The rax tree is basically a B-tree and its depth is bounded by the sizeof of
80 * the key. Holding 6 bytes for expiration-time key is more than enough to represent
81 * unix-time in msec, and in turn the depth of the tree is limited to 6 levels.
82 * At a first glance it might look sufficient but we need take into consideration
83 * the heavyweight maintenance and traversal of each node in the B-tree.
84 *
85 * And so, we can further prune the tree such that holding keys with msec precision
86 * in the tree doesn't bring with it much value. The active-expiration operation can
87 * live with deletion of expired items, say, older than 1 sec, which means the size
88 * of time-expiration keys to the rax tree become no more than ~4.5 bytes and we
89 * also get rid of the "noisy" bits which most probably will cause to yet another
90 * branching and modification of the rax tree in case of items with time-expiration
91 * difference of less than 1 second. The lazy expiration will still be precise and
92 * without compromise on accuracy because the exact expiration-time is kept
93 * attached as well to each item, in `ExpireMeta`, and each traversal of item with
94 * expiration will behave as expected down to the msec. Take care to configure
95 * `EB_BUCKET_KEY_PRECISION` according to your needs.
96 *
97 * EBUCKET KEY
98 * -----------
99 * Taking into account configured value of `EB_BUCKET_KEY_PRECISION`, two items
100 * with expiration-time t1 and t2 will be considered to have the same key in the
101 * rax-tree/buckets if and only if:
102 *
103 * EB_BUCKET_KEY(t1) == EB_BUCKET_KEY(t2)
104 *
105 * EBUCKETS CREATION
106 * -----------------
107 * To avoid the cost of allocating rax data-structure for only few elements,
108 * ebuckets will start as a simple linked-list and only when it reaches some
109 * threshold, it will be converted to rax.
110 *
111 * TODO
112 * ----
113 * - ebRemove() optimize to merge small segments into one segment.
114 * - ebAdd() Fix pathological case of cascade addition of items into rax such
115 * that their values are smaller/bigger than visited extended-segment which ends
116 * up with multiple segments with a single item in each segment.
117 */

128/*
129 * EB_BUCKET_KEY_PRECISION - Defines the number of bits to ignore from the
130 * expiration-time when mapping to buckets. The higher the value, the more items
131 * with similar expiration-time will be aggregated into the same bucket. The lower
132 * the value, the more "accurate" the active expiration of buckets will be.
133 *
134 * Note that the accurate time expiration of each item is preserved anyway and
135 * enforced by lazy expiration. It only impacts the active expiration that will
136 * be able to work on buckets older than (1<<EB_BUCKET_KEY_PRECISION) msec ago.
137 * For example if EB_BUCKET_KEY_PRECISION is 10, then active expiration
138 * will work only on buckets that already got expired at least 1sec ago.
139 *
140 * The idea of it is to trim the rax tree depth, avoid having too many branches,
141 * and reduce frequent modifications of the tree to the minimum.
142 */
143#define EB_BUCKET_KEY_PRECISION 0 /* TBD: modify to 10 */
144
145/* From expiration time to bucket-key */
146#define EB_BUCKET_KEY(exptime) ((exptime) >> EB_BUCKET_KEY_PRECISION)

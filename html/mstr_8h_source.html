9 * WHAT IS MSTR (M-STRING)?
10 * ------------------------
11 * mstr stands for immutable string with optional metadata attached.
12 *
13 * sds string is widely used across the system and serves as a general purpose
14 * container to hold data. The need to optimize memory and aggregate strings
15 * along with metadata and store it into Redis data-structures as single bulk keep
16 * reoccur. One thought might be, why not to extend sds to support metadata. The
17 * answer is that sds is mutable string in its nature, with wide API (split, join,
18 * etc.). Pushing metadata logic into sds will make it very fragile, and complex
19 * to maintain.
20 *
21 * Another idea involved using a simple struct with flags and a dynamic buf[] at the
22 * end. While this could be viable, it introduces considerable complexity and would
23 * need maintenance across different contexts.
24 *
25 * As an alternative, we introduce a new implementation of immutable strings,
26 * with limited API, and with the option to attach metadata. The representation
27 * of the string, without any metadata, in its basic form, resembles SDS but
28 * without the API to manipulate the string. Only to attach metadata to it. The
29 * following diagram shows the memory layout of mstring (mstrhdr8) when no
30 * metadata is attached:
31 *
32 * +----------------------------------------------+
33 * | mstrhdr8 | c-string | |
34 * +--------------------------------+-------------+
35 * |8b |2b |1b |5b |?bytes |8b|
36 * | Len | Type |m-bit=0 | Unused | String |\0|
37 * +----------------------------------------------+
38 * ^
39 * |
40 * mstrNew() returns pointer to here --+
41 *
42 * If metadata-flag is set, depicted in diagram above as m-bit in the diagram,
43 * then the header will be preceded with additional 16 bits of metadata flags such
44 * that if i'th bit is set, then the i'th metadata structure is attached to the
45 * mstring. The metadata layout and their sizes are defined by mstrKind structure
46 * (More below).
47 *
48 * The following diagram shows the memory layout of mstr (mstrhdr8) when 3 bits in mFlags
49 * are set to indicate that 3 fields of metadata are attached to the mstring at the
50 * beginning.
51 *
52 * +-------------------------------------------------------------------------------+
53 * | METADATA FIELDS | mflags | mstrhdr8 | c-string | |
54 * +-----------------------+--------+--------------------------------+-------------+
55 * |?bytes |?bytes |?bytes |16b |8b |2b |1b |5b |?bytes |8b|
56 * | Meta3 | Meta2 | Meta0 | 0x1101 | Len | Type |m-bit=1 | Unused | String |\0|
57 * +-------------------------------------------------------------------------------+
58 * ^
59 * |
60 * mstrNewWithMeta() returns pointer to here --+
61 *
62 * mstr allows to define different kinds (groups) of mstrings, each with its
63 * own unique metadata layout. For example, in case of hash-fields, all instances of
64 * it can optionally have TTL metadata attached to it. This is achieved by first
65 * prototyping a single mstrKind structure that defines the metadata layout and sizes
66 * of this specific kind. Now each hash-field instance has still the freedom to
67 * attach or not attach the metadata to it, and metadata flags (mFlags) of the
68 * instance will reflect this decision.
69 *
70 * In the future, the keys of Redis keyspace can be another kind of mstring that
71 * has TTL, LRU or even dictEntry metadata embedded into. Unlike vptr in c++, this
72 * struct won't be attached to mstring but will be passed as yet another argument
73 * to API, to save memory. In addition, each instance of a given mstrkind can hold
74 * any subset of metadata and the 8 bits of metadata-flags will reflect it.
75 *
76 * The following example shows how to define mstrKind for possible future keyspace
77 * that aggregates several keyspace related metadata into one compact, singly
78 * allocated, mstring.
79 *
80 * typedef enum HkeyMetaFlags {
81 * HKEY_META_VAL_REF_COUNT = 0, // refcount
82 * HKEY_META_VAL_REF = 1, // Val referenced
83 * HKEY_META_EXPIRE = 2, // TTL and more
84 * HKEY_META_TYPE_ENC_LRU = 3, // TYPE + LRU + ENC
85 * HKEY_META_DICT_ENT_NEXT = 4, // Next dict entry
86 * // Following two must be together and in this order
87 * HKEY_META_VAL_EMBED8 = 5, // Val embedded, max 7 bytes
88 * HKEY_META_VAL_EMBED16 = 6, // Val embedded, max 15 bytes (23 with EMBED8)
89 * } HkeyMetaFlags;
90 *
91 * mstrKind hkeyKind = {
92 * .name = "hkey",
93 * .metaSize[HKEY_META_VAL_REF_COUNT] = 4,
94 * .metaSize[HKEY_META_VAL_REF] = 8,
95 * .metaSize[HKEY_META_EXPIRE] = sizeof(ExpireMeta),
96 * .metaSize[HKEY_META_TYPE_ENC_LRU] = 8,
97 * .metaSize[HKEY_META_DICT_ENT_NEXT] = 8,
98 * .metaSize[HKEY_META_VAL_EMBED8] = 8,
99 * .metaSize[HKEY_META_VAL_EMBED16] = 16,
100 * };
101 *
102 * MSTR-ALIGNMENT
103 * --------------
104 * There are two types of alignments to take into consideration:
105 * 1. Alignment of the metadata.
106 * 2. Alignment of returned mstr pointer
107 *
108 * 1) As the metadatas layout are reversed to their enumeration, it is recommended
109 * to put metadata with "better" alignment first in memory layout (enumerated
110 * last) and the worst, or those that simply don't require any alignment will be
111 * last in memory layout (enumerated first). This is similar the to the applied
112 * consideration when defining new struct in C. Note also that each metadata
113 * might either be attached to mstr or not which complicates the design phase
114 * of a new mstrKind a little.
115 *
116 * In the example above, HKEY_META_VAL_REF_COUNT, with worst alignment of 4
117 * bytes, is enumerated first, and therefore, will be last in memory layout.
118 *
119 * 2) Few optimizations in Redis rely on the fact that sds address is always an odd
120 * pointer. We can achieve the same with a little effort. It was already taken
121 * care that all headers of type mstrhdrX has odd size. With that in mind, if
122 * a new kind of mstr is required to be limited to odd addresses, then we must
123 * make sure that sizes of all related metadatas that are defined in mstrKind
124 * are even in size.
125 */
